## 🔍 关键差异分析

### 1. RemoveAll方法

**AGFS实现**:
```go
func (fs *SomeFS) RemoveAll(path string) error {
    // 递归删除目录及其所有内容
    return fs.RemoveAll(path)
}
```

**EVIF状态**: ⚠️ **未实现**

**影响**:
- 无法原子化地删除整个目录树
- 需要通过多次`remove`调用实现

**优先级**: **高** - 这是一个重要的文件系统操作

---

### 2. 文件句柄 (Open/OpenWrite)

**AGFS设计**:
```go
type FileSystem interface {
    Open(path string) (io.ReadCloser, error)
    OpenWrite(path string) (io.WriteCloser, error)
}
```

**EVIF设计**:
- 不需要单独的`Open`方法
- 直接使用`read`返回`Vec<u8>`
- Rust的`Vec<u8>`比文件句柄更灵活

**优势**:
- ✅ 更简洁的API
- ✅ 无需管理文件句柄生命周期
- ✅ 更容易测试

**结论**: EVIF的设计**更优**,不需要实现Open/OpenWrite

---

### 3. 权限管理 (Chmod)

**AGFS**:
```go
Chmod(path string, mode uint32) error
```

**EVIF**: ⚠️ 未实现

**影响**: 低
- 大多数插件(LocalFS, MemFS)不需要运行时权限修改
- 可以在创建时指定权限

**优先级**: **低**

## 🎯 实现建议

### 高优先级 (建议实现)

#### 1. RemoveAll方法

**为什么重要**:
- 递归删除是常见操作
- 原子化删除更安全

**实现方案**:
```rust
async fn remove_all(&self, path: &str) -> EvifResult<()> {
    // 1. 检查是否为目录
    let info = self.stat(path).await?;

    if info.is_dir {
        // 2. 递归删除子项
        let children = self.readdir(path).await?;
        for child in children {
            let child_path = format!("{}/{}", path.trim_end_matches('/'), child.name);
            self.remove_all(&child_path).await?;
        }
    }

    // 3. 删除自身
    self.remove(path).await
}
```

**预计工作量**: ~50行代码/插件

---

### 中优先级 (可选实现)

#### 2. Touch方法

**用途**: 更新文件修改时间或创建空文件

**实现方案**:
```rust
async fn touch(&self, path: &str) -> EvifResult<()> {
    match self.stat(path).await {
        Ok(_) => {
            // 文件存在,更新时间戳(如果插件支持)
            Ok(())
        }
        Err(EvifError::NotFound(_)) => {
            // 文件不存在,创建空文件
            self.create(path, 0o644).await
        }
        Err(e) => Err(e)
    }
}
```

---

### 低优先级 (暂不需要)

#### 3. Chmod - 运行时权限修改
- 大多数场景不需要

#### 4. Symlink - 符号链接
- 复杂度高
- 安全风险
- 大多数文件系统不需要
